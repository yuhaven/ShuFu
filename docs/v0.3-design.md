# ShuFu v0.3 设计

> 状态：已实现参考版本  
> 日期：2026-06-21  
> 继承：v0.1 Windows/Linux 最小闭环、v0.2 Android/增量同步

## 1. 版本目标

v0.3 完成路线图中预定的 ESP32 接入，并解决低内存客户端面对长输出和大产物时的传输问题：

1. ESP32 通过 ESP-IDF SDK 调用 ShuFu Node；
2. Node、Python Client、CLI 和 ESP32 支持流式响应；
3. 产物支持内联、分块与显式外部引用；
4. 同步分别维护“本地已推送”和“远端已拉取”游标；
5. ESP32 只运行固件预注册 GPIO/传感器工具，不运行模型生成代码；
6. v0.1/v0.2 端点及 Bundle Schema 1/2 继续工作。

产品仍只解决一个问题：让不同硬件以一致、安全且足够轻量的方式获得大模型能力和连续上下文。v0.3 不扩张为 IoT 平台、设备管理后台或自动化编排系统。

## 2. 非目标

- 不在 ESP32 本地运行 LLM、GGUF 或向量模型；
- 不提供设备云、MQTT Broker、OTA、账号、RBAC 或证书中心；
- 不允许模型下发 Shell、脚本、动态库或可执行代码；
- 不把发现广播当作身份认证；
- 不在 v0.3 实现自主 Agent 循环；
- 不承诺公网明文 HTTP 部署安全；
- 不自动解析任意外部 URL，避免 SSRF 与凭据泄露。

## 3. 架构

```text
ESP32 firmware                       Windows/Linux ShuFu Node
┌────────────────────┐              ┌──────────────────────────┐
│ shufu_client       │──HTTP/NDJSON─>│ v3 invoke stream         │
│ bounded parser     │              │ Runtime + MemoryStore    │
│ firmware tools     │<─JSON events─│ Schema 1/2/3 sync        │
└────────────────────┘              └──────────────────────────┘
        │                                      ▲
        └─ GPIO / sensor, host-approved        │ v1/v2 compatible
                                               │
                                      Python CLI / Android v0.2
```

ESP32 是“远程神经末梢”：负责采集、展示和受控执行，模型推理由 Node 完成。网络连接、Wi-Fi 重连、Token 安全存储和 UI 确认由宿主固件负责。

## 4. 流式调用

`POST /shufu/v3/invoke/stream` 返回 `application/x-ndjson`。每行是一个完整 JSON 对象，事件顺序固定为：

```text
start -> delta* -> done
                 \-> error
```

每个事件包含稳定 `request_id` 和从 0 开始连续递增的 `sequence`。客户端必须逐行消费、限制单行大小、拒绝 request ID 变化或序号跳跃。ESP32 解析器使用固定 1024 字节缓冲，不累积完整响应。

当前 Runtime 抽象仍是阻塞式 `generate()`，因此参考 Node 在模型生成结束后按字符块发送 delta。该实现先稳定传输语义和边缘端内存边界；未来 Runtime 可提供 token 级生成，而无需改变 v0.3 线路协议。

## 5. Memory Schema 3

Schema 3 保留 session、message、artifact 不可变 ID、SHA-256 与 `changes.seq`。新增三种产物表示：

| 模式 | 字段 | 使用场景 |
| --- | --- | --- |
| `inline` | `content_base64` | 小文件或旧式一次性传输 |
| `chunks` | `content_chunks[]` | 默认大文件传输；每块独立校验 |
| `external` | `content_ref` | 先同步元数据，再从同一 Node 显式获取字节 |

`auto` 在 256 KiB 以下使用内联，以上使用 64 KiB 分块。每块携带 index、offset、size、SHA-256 和 Base64 内容；导入同时校验块序、块哈希、总大小与总哈希。

外部引用为同一 Node 下的相对路径 `/shufu/v3/artifacts/{id}/content`。Python Client 只解析回当前 Node 的引用，并重新校验大小和 SHA-256。MemoryStore 没有显式 resolver 时拒绝导入外部引用。该端点支持单个 HTTP `Range`，便于低内存设备按段读取。

## 6. 双游标同步

每个远端 Node 单独保存：

- `pushed_cursor`：本地变更已经被该远端确认到哪里；
- `pulled_cursor`：该远端变更已经被本地导入到哪里。

`POST /shufu/v3/sync/exchange` 在一次请求中携带本地增量并拉取远端增量。响应中的 `acknowledged_push_cursor` 只能推进 pushed cursor，`pull_bundle.cursor` 只能推进 pulled cursor；二者均单调不回退。

`changes.origin_node_id` 标记导入对象来源。为某个远端导出时排除来自该远端的对象，避免无意义回声；不可变 ID 与幂等导入仍是最终防重边界。v0.2 `/sync/pull` 始终返回 Schema 2，旧调用不受 Schema 3 默认策略影响。

## 7. ESP32 工具边界

`shufu_tool_registry_t` 最多注册 16 个编译进固件的 C 函数：

- 工具名最大 31 字符；
- 没有动态注册网络代码、Shell、`dlopen`、解释器或 `eval`；
- 传感器读取可标记无副作用；
- GPIO 写入、继电器、执行器必须标记 `has_side_effect=true`；
- 每次执行都必须传入 `allow_side_effect=true`，许可不缓存；
- 模型输出只是数据，宿主负责 JSON Schema 校验、用户确认和审计。

## 8. 兼容性

- `/shufu/v1/invoke`、v1 artifact/memory 端点保持不变；
- `/shufu/v1/memory/export` 继续返回 Schema 1；
- `/shufu/v2/sync/pull`、`push` 继续接收/返回 Schema 2；
- `MemoryStore.export_bundle()` 的 Python 默认仍为 Schema 2；
- v0.3 调用通过 `target_schema=3` 或 v3 端点显式升级；
- v1 capabilities 的历史 `protocol_version=0.2` 保留，同时新增 `latest_protocol_version=0.3` 和版本列表。

## 9. 验收条件

- Python Client/CLI 能重建完整流输出并验证事件顺序；
- Schema 3 分块 round-trip、损坏检测、外部引用显式解析通过；
- 双游标独立持久化、单调推进并排除来源回声；
- v0.1/v0.2 全部回归继续通过；
- ESP32 SDK 不含本地模型入口或动态执行原语；
- 纯 C 分帧/工具边界及 ESP 依赖客户端通过严格语法编译；
- 未做的真实 ESP-IDF 构建、Wi-Fi 压测与开发板验证必须明确披露。

## 10. 后续版本关系

v0.4 在本版本稳定的工具边界之上实现 Agent Lite、用户选择的产物注入和摘要记忆。v0.3 不提前引入自主循环，确保 ESP32 接入和传输协议可以独立审计。
