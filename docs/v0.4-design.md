# ShuFu v0.4 设计：Agent Lite 与安全上下文

## 1. 版本目标

v0.4 严格落实最初路线图中的三件事：

1. 在宿主注册的工具白名单上运行轻量、有限步骤的 Agent Lite；
2. 只有用户显式选择的 artifact 才能作为参考数据进入本次上下文；
3. 摘要记忆与原始消息分开保存，并能追溯到每一条来源消息。

它不是通用自主智能体平台。ShuFu 仍只解决“让不同硬件与系统安全调用模型”的单一问题；Agent Lite 是这个调用层上的小型能力，不引入插件市场、任意代码执行、远程桌面或后台自治任务。

## 2. 组件边界

```text
Host UI / API
  ├─ explicit artifact IDs
  ├─ side-effect approval callback
  └─ cancellation token
             │
       ContextBuilder ───── SummaryStore (derived DB)
             │                     │ source IDs + SHA-256
             │                     └──────────────┐
             └──── AgentContext ───────────────┐  │
                                               ▼  ▼
RuntimePlanner ── strict JSON decision ── AgentLite loop
                                               │
                                      host ToolRegistry
                                               │
                                      registered handlers only
```

模块职责：

| 模块 | 职责 | 明确不负责 |
| --- | --- | --- |
| `agent.py` | 工具白名单、工具元数据、硬限制 | 规划、动态加载工具 |
| `agent_lite.py` | 有界循环、审批、取消、审计、严格 JSON planner | Shell、`eval`、下载或执行代码 |
| `context.py` | 原始消息、摘要、artifact 的分层上下文 | 自动选择文档、修改原始记忆 |
| `summary.py` | 摘要独立持久化、来源指纹、长期事实策略 | 自动把推断写成事实 |

## 3. Agent Lite 状态机

每次 `run` 固定一个宿主创建的 `AgentLimits`，planner 无法在运行中提高限制。

```text
START
  │
  ├─ cancelled / deadline ──────────────> TERMINAL
  ▼
PLANNER DECISION (one strict JSON object)
  ├─ final ─────────────────────────────> COMPLETED
  ├─ unknown tool ── audit + observation ─┐
  └─ registered tool                      │
        ├─ side effect? ── approval        │
        │     ├─ deny ─ audit + observation│
        │     └─ allow once                │
        └─ execute + bounded observation ──┘
                 │
                 ├─ max steps ───────────> MAX_STEPS
                 ├─ deadline ────────────> TIMED_OUT
                 └─ cancel ──────────────> CANCELLED
```

默认上限是 3 步、10 秒；允许的宿主配置范围为 1–10 步、>0–300 秒。planner、等待审批和只读工具使用同一严格 deadline。已批准的副作用改为同步执行：如果 handler 超过 deadline，控制器会在它收敛后返回 `timed_out`，而不会先返回终态再让写操作留在后台。宿主仍必须为设备/网络 handler 配置内部超时。

## 4. Planner 与任意代码隔离

`RuntimePlanner` 复用现有 `Runtime.generate`，但只接受两种 JSON 数据：

- `{"action":"tool", ...}`；
- `{"action":"final", ...}`。

任何 Markdown 代码块、Python、JavaScript、Shell、表达式或未知 action 都只会触发解析错误。实现中没有 `eval`、`exec`、`subprocess`、动态 import 或把模型输出映射为文件名执行的路径。工具只能按稳定名称从宿主已经构造好的 `ToolRegistry` 取得。

## 5. 工具、副作用与逐次审批

`Tool.side_effect=True` 表示设备控制、文件写入、网络发送等可能改变外部状态的操作。每次具体调用都生成唯一 `ApprovalRequest.id`，并携带：

- step；
- 工具名称和说明；
- 本次参数快照。

只有本次回调返回的对象严格 `is True` 才将 `allow_side_effect=True` 传给 registry；`"true"`、`1`、非空 object 等均拒绝。参数先规范化为有限 JSON 深拷贝，审批 UI 和实际 handler 使用互不影响的副本，从而消除嵌套参数在批准后的竞态。批准不缓存、不跨步骤复用，也不能由 planner 自行声明。未配置审批回调等价于拒绝。

只读工具可提供 `cancel_handler`，在取消或超时时协作停止。副作用不放入 daemon worker；批准后在当前控制流内完成，终态不会早于副作用结束。取消不能回滚已经批准并开始的物理动作，因此 handler 应当短小、幂等并有自己的设备/网络超时。

## 6. 审计模型

一个 run 始终生成 `run_started` 和一个终止事件，中间可能包含：

- `tool_requested` / `tool_rejected`；
- `approval_requested` / `approval_granted` / `approval_denied`；
- `tool_completed` / `tool_failed` / `tool_cancelled`；
- `final_answer` / `run_error`。

每个事件含 run ID、事件 ID、UTC 时间、step 和有界详情。详情以 canonical JSON 保存，读取时返回副本，sink 无法回写历史。结果对象本身保存完整事件序列；可选 `audit_sink` 用于宿主持久化。CLI 使用 `JsonlAuditSink` 并对 token、密码、密钥、prompt、content、input 和 output 字段脱敏。自定义宿主 sink 仍需定义自己的隐私策略。

## 7. Artifact 上下文边界

Artifact 默认一个也不加载。调用方必须在本次 `ContextBuilder.build` 中传入 `selected_artifact_ids`。加载前依次验证：

1. ID 不重复，总数不超过 4；
2. artifact 属于当前 session；
3. MIME 在文本白名单中；
4. 单个不超过 64 KiB，总计不超过 128 KiB；
5. 内容长度与元数据一致，SHA-256 一致；
6. 内容是有效 UTF-8。

artifact 被渲染为 `role=user` 的 `[SHUFU_ARTIFACT_DATA]` 数据块，永远不拼接到 system 消息。固定 system policy 同时声明这类块是不可信参考数据。该设计降低提示注入权限，但不能声称模型层面的提示注入已被数学消除；高风险工具仍由白名单和用户审批形成确定性安全边界。

## 8. 摘要记忆

`SummaryStore` 使用独立的 `summaries.sqlite3`，不会改写 `MemoryStore` 的 `messages` 表。每个摘要保存：

- 摘要 ID、session、正文、创建时间；
- 每个来源原始消息的 ID、role、时间与内容 SHA-256；
- 可选长期事实数组。

`SummaryStore` 必须绑定原始 `MemoryStore`。调用者只提交消息 ID；Store 按 ID 读取真实消息、核对 session 并生成哈希。摘要读取及进入上下文前会再次复核来源仍存在、session/role/hash 未变。摘要 ID 必须显式选择，数量和正文总量受限，并以 user-role 派生数据块呈现。

默认 `allow_long_term_facts=False`，传入任何长期事实都会拒绝。只有宿主显式创建允许事实的 `SummaryPolicy` 才能保存，避免一次模型错误被默认为长期真相。

## 9. 兼容性

- `Tool` 既有的四个字段与 `ToolRegistry.execute(... allow_side_effect=...)` 行为保持兼容；
- v0.1/v0.2 Node 默认推理路径不自动启用 Agent；
- 原始 Memory Bundle 不自动携带摘要，避免旧端把派生信息误当原文；
- v0.4 组件可在 Windows/Linux Python 宿主直接使用，Android/ESP32 仍通过宿主协议适配，不在设备端执行 Python Agent。

## 10. 主程序集成状态

1. capabilities 已宣告 `agent_lite`、显式 artifact 上下文、独立 summary memory，并明确 `http_transport=false`；
2. `shufu agent` 显示工具名称、参数和副作用说明，每次调用单独询问；
3. CLI 只接收用户显式传入的 `--artifact-id` / `--summary-id`；
4. 审计写入 `SHUFU_HOME/audit/agent-runs.jsonl` 并执行字段脱敏；
5. `shufu summary` 通过真实 message ID 管理派生摘要；
6. HTTP Agent 仍不开放；后续若实现必须采用 [protocol-v0.4.md](protocol-v0.4.md) 数据契约，不允许远程注册 handler。
