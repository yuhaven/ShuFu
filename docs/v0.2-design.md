# ShuFu v0.2 设计

## 1. 版本目标

v0.2 把 v0.1 的桌面参考实现扩展为第一条真实跨设备链路：

```text
Windows / Linux ShuFu Node
        ⇅ invoke + sync
Android ShuFu App / AAR
        ⇣
Android 本地 GGUF 小模型
```

验收范围：

- Android App 和可复用 AAR；
- 发现局域网桌面 Node；
- 调用远程模型；
- 同步 session、消息和 Markdown/文档产物；
- Android 本地运行 GGUF 模型；
- v0.1 API 和 Bundle 保持可用。

ESP32 仍属于 v0.3。

## 2. 桌面 Node v0.2

桌面新增：

- UDP 7879 广播发现；
- 稳定 `node_id`；
- `changes` 单调序列；
- Bundle Schema 2；
- `/shufu/v2/sync/pull` 与 `/shufu/v2/sync/push`；
- `shufu discover` 与 `shufu sync`。

v0.1 端点继续提供 Schema 1 Bundle，避免旧客户端被强制升级。

## 3. Android SDK

### 3.1 网络调用

`ShuFuHttpClient` 只依赖 Android/JDK 标准网络能力，提供：

- `capabilities()`；
- `invoke()`；
- `pullSync()`；
- `pushSync()`。

不引入 Retrofit、协程或 DI 框架，宿主应用可自行决定线程模型。

### 3.2 节点发现

`ShuFuDiscoveryClient` 向 UDP 7879 广播 `SHUFU_DISCOVER_V2`，桌面返回节点 URL、名称、ID 和协议版本。

选择 UDP 广播而非引入第三方 mDNS 库，是为了保持 v0.2 依赖极少。后续可以在不改变 Node 语义的情况下增加 mDNS Adapter。

### 3.3 Android 记忆

`ShuFuMemoryStore` 使用 SQLite 和应用私有文件目录：

- 会话与消息进入 SQLite；
- 文档内容按 SHA-256 存储；
- 产物元数据保留文件名、MIME 和 session；
- change cursor 支持增量拉取；
- 导入通过对象 ID 和内容哈希幂等去重。

`ShuFuSyncEngine` 的 v0.2 策略：

1. 推送本地完整 Bundle；
2. 根据该远端节点的 cursor 拉取增量；
3. 导入本地并保存新 cursor。

完整推送虽然不如双游标高效，但逻辑简单、可恢复、容易诊断。消息和产物是不可变对象，重复推送不会重复插入。v0.3 可增加本地确认游标。

## 4. Android 本地模型

`LlamaCppRuntime` 通过 JNI 调用官方 llama.cpp `b9722`：

- 固定发布标签与归档 SHA-256；
- NDK r27c；
- ARM64；
- CPU 推理；
- GGUF 文件由用户导入或下载；
- 模型不会打包进 APK。

`ModelManager` 负责：

- 从 Android 文档选择器导入 GGUF；
- 从 HTTP(S) 下载；
- `.part` 临时文件；
- 可选 SHA-256 校验；
- 应用私有目录模型列表。

v0.2 使用贪心采样，优先验证稳定运行。温度、top-p、聊天模板和流式 token 属于后续优化。

## 5. App 体验

App 保持一个页面：

- Node URL；
- 可选 Token；
- Session；
- 输入框；
- 发现节点；
- 同步记忆；
- 调用 Node；
- 导入 GGUF；
- Android 本地生成。

本地生成写入同一个 Android MemoryStore，随后可以同步回 Windows。

## 6. 权限与风险

仍不引入账号、RBAC 或证书中心。

- 桌面默认仅监听回环地址；
- 局域网服务必须显式 `--allow-lan`；
- Token 可选；
- App v0.2 允许局域网明文 HTTP，便于开发与家庭网络验证；
- 面向互联网部署不是 v0.2 的目标。

主要风险：

| 风险 | v0.2 处理 |
| --- | --- |
| 广播伪造节点 | UI 显示 URL；敏感环境使用 Token 和手工 URL |
| HTTP 被监听 | 仅定位可信局域网；后续增加 TLS Adapter |
| Bundle 过大 | 拉取支持 cursor；大文件分块延后 |
| 双端冲突 | 消息/产物不可变 ID；不覆盖已有对象 |
| 模型来源不可信 | 支持 SHA-256；不自动执行或自动加载下载内容 |
| Android 内存不足 | 用户主动选择 GGUF；加载失败明确返回，不后台重启 |

## 7. 已完成验证

- Python v0.2 单元测试；
- Android JVM 单元测试；
- Kotlin SDK ↔ 真实 Python Node 跨语言测试；
- NDK ARM64 llama.cpp 和 JNI 编译；
- Debug AAR 构建；
- 可签名安装的 Debug APK 构建。

没有连接物理 Android 设备，因此本轮未对具体手机的推理速度、峰值内存和厂商兼容性作结论。
